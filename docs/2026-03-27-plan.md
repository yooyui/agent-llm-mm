# agent_llm_mm 规划（2026-03-27）

> 说明：本文档保留为 `2026-03-27` 的阶段规划记录。后续已有部分事项推进落地，例如配置文件驱动的 `openai-compatible` provider 集成；因此这里的“近期 / 中期 / 后期”应视为历史规划快照，而不是当前唯一 roadmap。当前路线图请以 [roadmap.md](roadmap.md) 为准。

## 1. 背景

当前仓库已经完成了 self-agent memory 的最小工程闭环，并具备本机 MCP `stdio` 接入能力。按当前实现复核结果看，项目的阶段性定位更准确地说是：

- 工程 MVP 闭环：已完成
- 本机 MCP 接入闭环：已完成
- 完整产品语义：部分完成

因此，今天的规划不再围绕“先把骨架搭出来”，而是围绕三个更现实的问题展开：

1. 如何把当前 MVP 从“可运行”推进到“可稳定接入”
2. 如何把最小 memory / reflection 语义推进到更完整、但不过度设计的版本
3. 如何给后续真实模型接入和 richer memory 体系预留干净演进路径

## 2. 当前状态

### 已实现

- `ingest_interaction` 已能记录事件并持久化派生命题
- `build_self_snapshot` 已能从持久化状态构建快照
- `run_reflection` 已支持显式 `replacement_evidence_event_ids`
- `namespace` 已在 domain / SQLite / MCP DTO 三层打通
- `doctor` / `serve` 已可作为本机接入入口
- MCP `stdio` 已暴露 4 个工具
- 自动化测试已覆盖关键链路，截至 `2026-03-27` 共 50 个测试通过

### 部分实现

- `decide_with_snapshot` 前半段 gate 为真实能力，后半段仍依赖 mock model
- `SnapshotBudget` 仍只控制 evidence 数量
- `episodes` 仍以 `episode_reference -> event_id` 关联为主
- 默认 SQLite 路径已有持久化，但作用域语义未定型

### 未实现

- 真实 LLM provider
- 自动 evidence lookup 与 richer evidence 语义
- reflection 对 `identity_core` 与 `commitments` 的深层修订
- richer `identity / claim / episode / reflection` schema
- working memory / procedural memory 独立建模

## 3. 近期规划

### 目标

把当前 MVP 从“可运行”推进到“可稳定接入、边界清楚、验证可重复”。

### 重点事项

1. 明确默认 SQLite 路径的作用域语义

当前实现最大的不确定性不是功能缺失，而是默认数据库到底应该：

- 按用户共享
- 按项目隔离
- 按 workspace 隔离

这个语义必须先定下来，否则文档、测试和接入方式会持续漂移。

2. 为默认数据库路径策略补自动化测试

需要用测试锁住默认路径行为，避免后续修改 `config`、脚本或 bootstrap 逻辑时产生无感回归。

3. 收口 release gate

把当前仓库的最低发布门槛固定为：

- `cargo fmt --check`
- `cargo clippy --all-targets --all-features -- -D warnings`
- `cargo test`
- `pwsh -File .\scripts\agent-llm-mm.ps1 doctor`

4. 明确能力承诺边界

需要正式区分：

- 可作为正式接入能力暴露的工具
- 仅用于调试、实验或流程验证的工具

特别是 `decide_with_snapshot`，当前不应被当成生产级决策能力承诺。

5. 补齐最小接入操作手册

围绕正式库、测试库、实验库的划分、并发访问注意事项、推荐数据库命名和基本排障路径，形成一套简单且可执行的约定。

### 近期完成标志

- 默认数据库作用域语义有明确结论
- 对应自动化测试已补齐
- 文档口径与实现边界不再冲突
- 新接入者可以按文档稳定完成一次 `doctor -> serve` 流程

## 4. 中期规划

### 目标

把当前“最小 evidence-aware reflection”推进为更完整的 memory / reflection 语义，但仍坚持 KISS 和 YAGNI，不提前进入完整产品化复杂度。

### 重点事项

1. 把 reflection 从“显式 evidence 输入”推进到“显式输入 + 可选查询”

当前调用方必须先知道 event id，这适合 MVP，但会限制中期可用性。中期需要保留显式输入分支，同时引入可选查询能力。

2. 抽象 evidence-oriented 查询接口

当前 `EventStore::has_event` 只够做存在性校验，不足以支撑：

- 独立证据条数判断
- 证据去重
- 更明确的 evidence 查询和筛选

3. 丰富 `episodes` 的结构表达

逐步把 episode 从引用层推进到轻量聚合层，优先考虑引入：

- `goal`
- `outcome`
- `lesson`

4. 丰富 `identity_core` 与 claim schema

当前字符串列表足够支撑 MVP，但不足以表达慢变量、自我修订和更明确的稳定性语义。中期应先补最有用、最容易验证的一层字段，而不是一次性铺满 schema。

5. 明确 reflection 策略边界

需要把“何时 dispute、何时 replacement、何时拒绝、何时需要更多证据”写成更明确的规则与测试，而不是只靠当前最小实现隐式表达。

### 中期完成标志

- reflection 不再只能依赖调用方手工传全量 evidence id
- evidence 查询接口已成型并有测试保护
- `episodes` 与 `identity_core` 不再只是占位表达
- 文档能清楚说明 reflection 的允许条件和拒绝条件

## 5. 后期规划

### 目标

从“工程化 MVP”逐步演进到“更完整的 self-agent memory 系统”。

### 重点事项

1. 接入真实 LLM provider

`decide_with_snapshot` 只有在接入真实模型后，才有可能从流程验证能力升级为真正的决策能力。

2. 引入更完整的多层 memory 体系

需要逐步补 working memory、procedural memory，以及层间筛选、回写和淘汰策略。

3. 推进 identity / commitment 深层修订机制

这是从“命题级反思”演进到“自我机制反思”的关键一步，但必须建立在 schema、evidence 和 reflection 策略已经足够稳定的前提下。

4. 评估 transport、配置与产品化边界

需要重新判断项目是否长期只保留本机 `stdio` 形态，是否需要更正式的配置层、更多 transport 或更清晰的数据隔离策略。

5. 评估 richer 审计与运营支持能力

包括更细的 reflection trace、决策来源说明和后续可能的辅助可视化能力，但这些都应排在核心语义稳定之后。

### 后期完成标志

- 决策链不再依赖 mock model
- memory 分层不再只是概念存在，而是有实际结构与回路
- reflection 可以开始影响更深层的 identity / commitment 语义
- 系统的接入边界、存储边界和产品边界都有明确定位

## 6. 建议执行顺序

建议按下面顺序推进，避免并行改太多基础语义导致说明、测试和实现同时漂移：

1. 先收口默认数据库作用域语义与自动化测试
2. 再固化 release gate 和接入操作手册
3. 然后推进 evidence-oriented 查询接口
4. 在 evidence 能力更稳定后，扩展 reflection 策略
5. 再逐步丰富 `episodes`、`identity_core` 和 richer schema
6. 最后再考虑真实模型接入与更完整的多层 memory

## 7. 规划原则

这份规划默认遵循以下原则：

- KISS：先解决真实不确定性，再扩展 richer 语义
- YAGNI：不提前实现当前没有明确验证价值的复杂机制
- SOLID：把配置、持久化、应用编排和策略判断分层清楚
- DRY：规则、文档、测试和接入方式都应尽量保持单一来源

当前阶段最重要的不是“把所有理想能力一次做完”，而是让系统的默认运行语义、接入边界和反思边界先稳定下来。
